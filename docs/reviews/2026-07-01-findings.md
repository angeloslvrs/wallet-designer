# Project Review Findings

Review date: 2026-07-01

Scope: code quality, project health, PassKit behavior, and release/verification posture for the current repository state. This review is based on local code inspection, local test/build runs, and Apple Wallet/PassKit documentation.

## Apple Documentation Referenced

- [Pass object reference](https://developer.apple.com/documentation/walletpasses/pass): `passTypeIdentifier` must match the signing certificate, and `passTypeIdentifier` plus `serialNumber` uniquely identify a pass.
- [Creating an airline boarding pass using semantic tags](https://developer.apple.com/documentation/walletpasses/creating-an-airline-boarding-pass-using-semantic-tags): iOS/watchOS 26 semantic boarding passes require `semanticBoardingPass` in `preferredStyleSchemes`, `boardingPass.transitType = PKTransitTypeAir`, and required semantic tags; Apple recommends keeping legacy fields for compatibility.
- [SemanticTags reference](https://developer.apple.com/documentation/walletpasses/semantictags): semantic tags are Apple-defined machine-readable metadata.
- [Register a pass for update notifications](https://developer.apple.com/documentation/walletpasses/register-a-pass-for-update-notifications): registration uses `POST /v1/devices/{deviceLibraryIdentifier}/registrations/{passTypeIdentifier}/{serialNumber}` with `Authorization: ApplePass {authenticationToken}` and returns `201` for a new registration or `200` if already registered.
- [Unregister a pass for update notifications](https://developer.apple.com/documentation/walletpasses/unregister-a-pass-for-update-notifications): unregister uses the same pass identity and `ApplePass` authorization.
- [Adding a web service to update passes](https://developer.apple.com/documentation/walletpasses/adding-a-web-service-to-update-passes) and the archived [PassKit update guide](https://developer.apple.com/library/archive/documentation/UserExperience/Conceptual/PassKit_PG/Updating.html): web-service URLs must be HTTPS in production, authentication tokens must remain stable, updates are keyed by pass type ID and serial number, APNs update payloads are empty JSON dictionaries, and servers should support `If-Modified-Since`.
- Archived [PassKit package creation guide](https://developer.apple.com/library/archive/documentation/UserExperience/Conceptual/PassKit_PG/Creating.html): the package manifest excludes `manifest.json` and `signature`, stores SHA-1 file hashes, and the signature is a detached PKCS#7 signature over the manifest using the Pass Type ID certificate plus WWDR certificate.
- [Sending notification requests to APNs](https://developer.apple.com/documentation/usernotifications/sending-notification-requests-to-apns): APNs uses HTTP/2 over TLS, requires headers such as `apns-topic` and `apns-push-type`, and callers should handle connection and status-code behavior.

## Findings

### P2 - Stored pass identity is serial-only, while Apple identity is pass type ID plus serial number

Apple defines a pass as uniquely identified by `passTypeIdentifier` and `serialNumber`, and the web-service routes include both values. Locally, the store treats `serial` as globally unique in multiple places:

- `apps/server/src/storage.js:38` defines `passes(serial TEXT PRIMARY KEY)`.
- `apps/server/src/storage.js:50` defines `registrations` as `PRIMARY KEY(device_library_identifier, serial)`, with `pass_type_identifier` stored but not part of the key.
- `apps/server/src/storage.js:180` upserts passes with `ON CONFLICT(serial)`.
- `apps/server/src/storage.js:219`, `apps/server/src/storage.js:221`, and `apps/server/src/storage.js:259` look up existing records by serial alone.
- `apps/server/src/storage.js:364` uses `INSERT OR REPLACE` for registrations keyed by device and serial.
- `apps/server/src/storage.js:372` unregisters by device and serial only.
- `apps/server/src/storage.js:377` joins updated serial queries by `p.serial = r.serial`.
- `apps/server/src/storage.js:417` builds the admin snapshot map by serial.
- `apps/server/src/routes/admin.js:211` has a comment acknowledging Apple keys passes by serial plus pass type, but the implementation still rejects or merges by serial alone.

Impact: the current production configuration likely masks this because `PASS_TYPE_ID` is effectively global. But if this project ever supports more than one pass type ID, imports historical data with a different pass type, or runs tests/dev fixtures that collide by serial, one pass can overwrite another and device registrations can become ambiguous. Push targeting, list-updated responses, unregister behavior, and admin views can all point at the wrong pass.

Suggested fix: choose an explicit invariant. If the product only supports one `PASS_TYPE_ID`, enforce that invariant at write time and fail loudly when an existing serial has a different pass type. If multiple pass types are in scope, migrate storage and route APIs to composite identity: `passes(pass_type_identifier, serial)`, `registrations(device_library_identifier, pass_type_identifier, serial)`, and update admin snapshots/device counts accordingly.

### P2 - Wallet registration always returns `201`, including repeat registrations

Apple documents `201` for a new device registration and `200` when the serial is already registered for that device. The current route always returns `201` after `registerDevice`:

- `apps/server/src/routes/wallet.js:23` defines the registration endpoint.
- `apps/server/src/routes/wallet.js:34` calls `store.registerDevice(...)`.
- `apps/server/src/routes/wallet.js:36` always sends `res.status(201).end()`.
- `apps/server/src/storage.js:364` uses `INSERT OR REPLACE`, so the store does not tell the route whether a row was newly created or already existed.

Impact: Wallet clients may tolerate this, but it is a spec mismatch and it hides idempotency behavior from tests. It also interacts poorly with the serial-only key above because a repeated registration for the same device and serial but a different pass type would replace the stored pass type.

Suggested fix: make `registerDevice` return whether it inserted a new row. Return `201` for inserted and `200` for already present. If fixing pass identity, include `pass_type_identifier` in the registration primary key before changing status handling.

### P2 - CI does not run the Node app regression gates

The repository has solid local gates, but the only GitHub Actions workflow is Apple validation:

- `.github/workflows/apple-validate.yml:31` runs Node `20`, while `package.json` requires Node `>=24` and `apps/server/src/storage.js` uses built-in `node:sqlite`.
- `.github/workflows/apple-validate.yml:38` builds only the `dev-sample` template pass.
- `.github/workflows/apple-validate.yml:56` runs Apple's `buildpass validate`.
- There is no workflow running `npm test`, `npm run check`, or `npm run build:designer`.

Impact: the Apple validation gate is valuable, but it covers package shape and validator compatibility for one template path. Server routes, SQLite storage, designer behavior, fixture validation, schema regressions, and FormState pass generation can regress without blocking merges.

Suggested fix: add a separate Node 24 CI workflow with `npm ci`, `npm test`, `npm run check`, `npm run build:designer`, and at least one FormState plus one template `npm run build:pass` command. Keep the Apple validator workflow as the PassKit-specific gate.

### P3 - The FormState schema is permissive where emitted pass data is forwarded to Apple

The schema is strict at the top level, but some nested structures are open or only loosely typed:

- `packages/pass-schema/schema.json:47` allows `semantics` as an unconstrained object.
- `packages/pass-schema/schema.json:63` defines `additionalInfoFields` items without `additionalProperties: false`.
- `packages/pass-schema/schema.json:89` and `packages/pass-schema/schema.json:102` define `upcomingPassInformation` and Wi-Fi item objects without strict property rejection.
- `packages/pass-schema/schema.json:116` defines display field objects without strict property rejection.
- `packages/pass-builder/form-to-pass.js:54` emits every semantics entry and falls back to `{ text: value }` for unknown objects.

Impact: typos in Apple semantic keys or field properties can pass local validation and only surface during Apple validation or on-device behavior. Apple's `SemanticTags` vocabulary is finite, and this codebase already maintains a boarding subset in `packages/pass-builder/semantics.js`, so the project has enough local knowledge to catch many of these mistakes earlier.

Suggested fix: add a schema-level or validator-level warning/error path for unknown semantic keys and unsupported nested field properties. If complete strictness would block future Apple keys, keep an explicit extension escape hatch and make normal fixture/designer fields strict.

### P3 - The designer bundle is large because all views and heavy scanner/barcode dependencies load eagerly

The production designer build succeeds, but Vite reports one JavaScript chunk at 1,459.76 kB minified and 396.54 kB gzip. The entry point imports all major views up front:

- `apps/designer/src/main.js:1` imports `renderForm`.
- `apps/designer/src/main.js:5` imports `mountIssue`.
- `apps/designer/src/main.js:6` imports `mountManage`.
- `apps/designer/src/issue.js:9` imports `bwip-js`.
- `apps/designer/src/issue.js:10` imports scanner helpers; `apps/designer/src/scan.js` imports `@zxing/browser`.
- `apps/designer/src/main.js:56` chooses the active tab at runtime, but the bundle has already pulled in all views.

Impact: this is acceptable on a private LAN tool today, but it increases first-load cost and makes the designer surface pay for issue/scanner dependencies even when users only manage passes or edit FormState data.

Suggested fix: dynamic-import view modules on tab activation and load scanning/barcode-generation libraries only when the relevant issue workflow needs them. This is a contained code-splitting improvement and does not require changing frameworks.

## Current Strengths

- Server-controlled identity overrides are centralized in `apps/server/src/pass-build.js:29` and `apps/server/src/pass-build.js:46`, so templates and FormState records cannot persistently override `serialNumber`, `authenticationToken`, `passTypeIdentifier`, `teamIdentifier`, or `webServiceURL`.
- Production `WEB_SERVICE_URL` is required to be HTTPS in `apps/server/src/env-validate.js:1`, matching Apple's production web-service requirement.
- Authentication token stability is intentional in `apps/server/src/storage.js:221` and `apps/server/src/storage.js:259`, which aligns with Apple's warning not to rotate installed-pass tokens.
- The package signing path in `packages/pass-builder/sign.js:21` follows Apple's manifest/signature model: SHA-1 manifest, detached PKCS#7 signature, WWDR certificate inclusion, and zip output.
- The semantic boarding pass implementation is directionally aligned with Apple's iOS/watchOS 26 docs: `packages/pass-builder/form-to-pass.js:17` sets `PKTransitTypeAir`, `packages/pass-builder/form-to-pass.js:27` includes `semanticBoardingPass`, and `packages/pass-builder/semantics.js:125` tracks required/recommended boarding semantics.
- Template handling has useful guardrails: unknown template data keys fail in `packages/pass-builder/template.js:134`, server identity is applied after template merge in `packages/pass-builder/template.js:247`, and missing base image variants are synthesized for emitted pass packages in `packages/pass-builder/template.js:262`.
- The web-service route supports `If-Modified-Since` in `apps/server/src/routes/wallet.js:72`, and APNs update pushes use HTTP/2, certificate auth, an empty JSON payload, `apns-topic`, and `apns-push-type` in `apps/server/src/apns.js:78`.
- The control-plane/public-plane split is explicit in `apps/server/src/index.js:17` and `apps/server/src/middleware/guard.js`, with no wildcard CORS added.

## Suggested Double-Check Tasks

1. Add or run a test that issues two records with the same serial and different pass type IDs, then verify whether the second write clobbers the first.
2. Add or run a wallet registration test that calls the registration endpoint twice and expects `201` then `200`.
3. Confirm whether the product intentionally supports exactly one production `PASS_TYPE_ID`; if yes, encode that as an invariant. If no, prioritize the composite-key migration.
4. Try a fixture with a misspelled semantic key such as `departureLocationTimezone` and confirm whether local validation catches it before Apple validation.
5. Consider a Node 24 CI workflow for `npm test`, `npm run check`, and `npm run build:designer` so non-Apple regressions are gated before merge.
