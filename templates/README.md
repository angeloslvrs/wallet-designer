# templates/

`.pkpasstemplate` bundles the server can issue passes from. Each bundle is a
directory named `<id>.pkpasstemplate` containing a skeleton `pass.json` plus
its images (`icon.png`, `logo.png`, optional `*.lproj/` localizations).

## Field keys are arbitrary; bindings reconcile them

No code assumes a template's field-key names. Apple's semantic vocabulary
(`departureGate`, `currentBoardingDate`, `transitStatus`, …) is fixed and
hardcoded in `packages/pass-builder/semantics.js`; each template's own field
keys are reconciled to it by a discovered `semanticKey → fieldKey` **binding
map** (`packages/pass-builder/bindings.js`), recomputed on every upload and
editable from the Issue tab's Templates card
(`PUT /api/templates/:id/bindings`). The two committed bundles deliberately use
different key conventions to exercise this:

- **`cebpac.pkpasstemplate`** — a real Pass Designer 1.0 export. Keys like
  `depart`, `arrive`, `date`, `boardingTime`, `term`, `sequence`.
- **`dev-sample.pkpasstemplate`** — a hand-written stand-in from before Pass
  Designer (macOS 27 beta) was available; its `pass.json` mirrors what
  `packages/pass-builder/form-to-pass.js` emits. Different keys
  (`gate`, `seat`, `passenger`, `flight`, `boarding`, `depart-time`, `group`,
  `seq`, `ff`, `terminal-dep`, `terminal-arr`) — both bind to the same
  semantics through discovery, no per-template code.

The status API speaks semantic keys on both shapes; a semantic with no bound
visible field still updates the pass (iOS 26 renders `semanticBoardingPass`
from semantics alone) and is reported in `skippedFields` as informational.

PNGs for both bundles are generated by `npm run init` (see
`scripts/gen-assets.js`) and are gitignored — only `pass.json` is committed.

## Why the bundles carry sample `semantics`

Each `pass.json` `semantics` block carries **sample values** (flight number,
boarding/departure/arrival dates, passenger name, seat, airport timezones and
city names) because Apple's `BoardingPassValidator` treats those as required —
the CI gate (`.github/workflows/apple-validate.yml`) builds `dev-sample` with
no per-pass data and fails on missing semantics. At issue time the volatile
placeholders (the schedule dates, `passengerName`, `seats`) are null-cleared
unless re-derived from bound per-pass data, and explicit `data.semantics`
deep-merges over the rest.

## Adding another template

1. Drop the bundle here as `<id>.pkpasstemplate/`, or upload it from the
   browser: SPA → **Issue** tab → Templates card (zip the bundle first). The
   curl equivalent:
   `zip -r flight.zip MyFlight.pkpasstemplate` then
   `curl --data-binary @flight.zip -H "Content-Type: application/zip" https://<host>/api/templates/<id>`
2. Upload (re-)discovers the bundle's bindings automatically. Review them in
   the Templates card and correct any wrong guesses
   (`PUT /api/templates/:id/bindings`) — no code change needed for a new key
   convention.
3. `GET /api/templates` lists installed templates with the field keys and
   discovered bindings each declares — that is the merge API for issuing passes
   against it. The Issue tab renders the same list, with per-template delete
   (`DELETE /api/templates/:id` — refused with 409 while any stored pass still
   references the bundle, because installed passes rebuild from it on every
   device fetch).

Server-controlled fields (`serialNumber`, `authenticationToken`,
`webServiceURL`, `passTypeIdentifier`, `teamIdentifier`) are always injected at
build time and never trusted from a template.
