# templates/

`.pkpasstemplate` bundles the server can issue passes from. Each bundle is a
directory named `<id>.pkpasstemplate` containing a skeleton `pass.json` plus
its images (`icon.png`, `logo.png`, optional `*.lproj/` localizations).

## dev-sample — a stand-in, not a Pass Designer export

`dev-sample.pkpasstemplate` was written by hand to unblock the template
pipeline before Pass Designer (macOS 27 beta) was available. Its `pass.json`
mirrors what `packages/pass-builder/form-to-pass.js` emits, and its field keys
(`gate`, `seat`, `passenger`, `flight`, `boarding`, `depart-time`, `group`,
`seq`, `ff`, `terminal-dep`, `terminal-arr`) are the de-facto key convention
the server's status routes assume (see `apps/server/src/template-status.js`).

Its PNGs are generated by `npm run init` (see `scripts/gen-assets.js`) and are
gitignored — only `pass.json` is committed.

## When a real Pass Designer export lands

1. **Inspect its `pass.json` first** — especially the field `key` names. The
   merge pipeline is convention-agnostic, but `template-status.js` hardcodes
   the keys above; adjust it if Designer emits different ones.
2. Drop the bundle here as `<id>.pkpasstemplate/`, or upload it from the Mac:
   `zip -r flight.zip MyFlight.pkpasstemplate` then
   `curl --data-binary @flight.zip -H "Content-Type: application/zip" https://<host>/api/templates/<id>`
3. `GET /api/templates` lists installed templates and the field keys each
   declares — that list is the merge API for issuing passes against it.

Server-controlled fields (`serialNumber`, `authenticationToken`,
`webServiceURL`, `passTypeIdentifier`, `teamIdentifier`) are always injected at
build time and never trusted from a template.
